# 1. Product Overview

## 1.1. 서비스 이름(가칭)

* **"VAKD Finder" – 일반 고등학생을 위한 학습 전략 설계 도구**

## 1.2. 목적

* 일반 고등학생(1학년~3학년)이 자신의 **정보 처리·학습 스타일(VAKD)** 을 이해하고,
  → 과목별·상황별로 **가장 효율적인 공부 전략을 스스로 설계**할 수 있도록 돕는 웹 서비스.

📌 핵심 가치

* ❌ 진로 결정 도구
* ❌ 단순 공부 팁 제공
* ✅ **전략 선택 능력 강화**
* ✅ **메타인지 능력 향상**

## 1.3. 핵심 컨셉

* 기존 종이 테스트로 사용되던 **"선호 표상 체계 검사 (Favored Representational Systems)"**를
  **웹 기반 자동 채점 + 학습 전략 설계 도구**로 재구현. 
* 결과는 **4가지 척도**로 제공:

  * V: Visual (시각형)
  * A: Auditory (청각형)
  * K: Kinesthetic (신체감각형)
  * D: Digital/Logical (논리형)

## 1.4. 일반 고등학생 결과지 설계 원칙

1. 유형은 '정체성'이 아니라 '전략 선택 기준'으로 설명한다.
2. 과목별 적용 사례를 반드시 포함한다.
3. 성적 문제를 노력이나 능력 문제가 아닌 전략 불일치로 해석한다.
4. 현재 학습과 고3 대비 전략을 구분해 제시한다.
5. 학생이 '스스로 조정할 수 있다'는 감각을 남긴다.

# 2. 목표 & 성공 지표

## 2.1. 비즈니스/교육 목표

* 고등학교 1~3학년 대상 수업·상담에서 활용 가능한 **무료 웹 진단 도구** 제공.
* 학생이 결과지를 보고 **"이 상황에서는 이 전략이 낫다"**를 스스로 판단할 수 있도록 돕는다.
* **"공부를 하고는 있는데, 제대로 하고 있는지 확신이 없는 상태"**에서 벗어나게 한다.

## 2.2. 성공 지표(예시)

* 1차 릴리즈 후:

  * 하루 평균 진단 완료 수 50건 이상
  * 검사 완료율(시작 대비 결과 페이지 도달 비율) 80% 이상
  * 결과 공유 버튼 클릭률 30% 이상

# 3. 타깃 유저

## 3.1. 1차 타깃

* **고등학교 1학년부터 3학년까지의 일반 고등학생**

  * 스마트폰으로 접속 (모바일 비중 80% 이상 가정)
  * 주요 고민: "왜 성적이 안 오르지?", "이 방법이 맞나?"
  * 시간 관점: **지금의 선택이 미래에 영향**을 준다는 인식
  * 비교 기준: 또래 성적, 상위권 학생의 공부법
  * 위험 요소: 잘못된 공부 습관의 고착화

## 3.2. 2차 타깃

* 담임교사, 진로진학 상담교사, 강사
  → 이후 "강사용 대시보드"는 차기 버전에서 고려 (이번 PRD 범위에는 **포함 X**, 단 데이터 구조는 확장 가능하게 설계)

# 4. 범위(Scope)

## 4.1. In Scope (v1)

1. **홈 페이지 + 검사 시작 화면**
2. **10문항 VAKD 검사 UI** (문항/선택지 텍스트 하드코딩 or JSON)
3. **원 테스트와 동일한 채점 로직**
4. **결과 페이지**

   * V/A/K/D 점수 표시 (그래프 포함)
   * 주/보조 유형 설명 (전략 선택 기준으로 설명)
   * **과목별 적용 가이드** (국어, 수학, 영어, 탐구 등)
   * **성적 정체 원인 분석** (전략 불일치 관점)
   * **지금 vs 앞으로 전략 구분** (현재 학습 / 고3 대비)
   * **자기조절 학습 팁** (계획, 집중, 복습 주기)
   * 결과 공유용 썸네일 이미지 생성(프론트에서 캡처)
5. **기초 로그 저장**

   * 익명 세션 ID, 점수, 검사 시간, 디바이스 타입
6. **기본 관리 기능**

   * (관리자 페이지 없음)
   * DB에 결과만 적재, 별도 UI 없이 추후 분석용

## 4.2. Out of Scope (v1)

* 로그인/회원제
* 선생님 계정/반별 관리
* 이메일 발송, 카카오 연동 등
* MBTI 등 다른 심리검사와의 결합
* 멀티 언어 지원 (v1은 **한국어 전용**)

# 7. 기능 상세

## 7.1. 홈 페이지 (`/`)

### 7.1.1. 콘텐츠

* 히어로 문구:

  * "공부는 하고 있는데, 제대로 하고 있는지 확신이 없나요?"
  * "VAKD 진단으로 **나에게 맞는 학습 전략**을 설계해보세요."
* 간단 설명:

  * V, A, K, D가 무엇인지 1줄씩 요약
* 버튼:

  * **[검사 시작하기]** → `/test`
* 하단:

  * 사용 시간 안내: “약 5~7분 소요”
  * “개인 정보는 저장하지 않습니다. (결과 통계만 익명 수집)”

### 7.1.2. UX 요구사항

* 모바일 퍼스트, 스크롤 길이 짧게
* CTA 버튼 뚜렷하게

## 7.2. 검사 페이지 (`/test`)

### 7.2.1. 검사 구조

* 총 **10문항**, 각 문항에 **4개의 진술문**.
* 사용자는 각 문항에서 다음을 수행:

  * **각 진술문에 4, 3, 2, 1 중 하나의 점수를 부여**
  * 단, **한 문항 내에서는 4/3/2/1을 각각 한 번씩만 사용**.
  * (원 검사 방식 그대로 구현 – 총점 100점) 

#### 7.2.1.1. 점수 부여 UX 안

* **선호안 A (라디오 매트릭스 방식)**

  * 각 진술문 오른쪽에 [4][3][2][1] 라디오 버튼
  * 한 문항 내에서 각 숫자는 한 번만 선택 가능하도록 제약 로직
  * 선택 충돌 시 이전 선택 해제

* **대체안 B (순위 선택 방식)**

  * 진술문 카드 4개를 “가장 나와 비슷한 것(4점) → 가장 덜 비슷한 것(1점)” 순으로 드래그 정렬
  * 정렬 완료 후 자동으로 4/3/2/1 매핑
  * 구현 난이도가 올라가므로 v1에서는 A안 권장

### 7.2.2. 문항/선택지 데이터 구조

* 예시(JSON):

```ts
type Question = {
  id: number; // 1~10
  title: string; // “1. 내가 중요한 결정을 할 때…”
  options: {
    id: number; // 1~4
    text: string; // 진술문 내용
    dimension: 'V' | 'A' | 'K' | 'D'; // 아래 7.3.3 참고
  }[];
};
```

* 모든 문항/선택지는 서버 또는 프론트에 상수로 저장 (v1 하드코딩 OK).

### 7.2.3. 진행 상태

* 상단에 progress bar: `문항 n / 10`
* “이전/다음” 버튼 (문항 이동)
* 모든 문항에 4/3/2/1이 모두 채워지기 전에는 **“제출하기” 비활성화**.

### 7.2.4. 유효성 검사

* 문항 기준:

  * 각 문항에 **4개의 점수(4,3,2,1) 모두 사용** 되었는지 확인.
* 전체 기준:

  * 모든 문항 유효 → 채점 및 결과 페이지 이동.

# 8. 채점 로직(Assessment Engine)

## 8.1. 기본 규칙

1. 각 문항에서 4·3·2·1을 진술문에 할당.
2. 각 진술문은 하나의 차원(V/A/K/D)에 매핑됨.
3. **각 차원의 총점 = 해당 차원에 속한 진술문에 부여된 점수의 합.**
4. 전체 합은 항상 100점이 되어야 함 (검산용). 

## 8.2. 문항별 옵션 → 차원 매핑

**행 = 선택지 번호(1~4), 열 = 문항 번호(1~10)**
(PDF의 채점표를 그대로 옮긴 것) 

### 8.2.1. 매핑 테이블

| 문항 | 옵션1 | 옵션2 | 옵션3 | 옵션4 |
| -- | --- | --- | --- | --- |
| 1  | K   | A   | V   | D   |
| 2  | A   | V   | D   | K   |
| 3  | V   | K   | D   | A   |
| 4  | A   | D   | K   | V   |
| 5  | A   | D   | K   | V   |
| 6  | K   | V   | A   | D   |
| 7  | A   | V   | D   | K   |
| 8  | V   | A   | K   | D   |
| 9  | A   | D   | K   | V   |
| 10 | V   | A   | K   | D   |

→ 개발 시, Question 옵션 객체에 `dimension` 필드로 이 값들을 설정.

## 8.3. 주/보조 유형 판정

1. V/A/K/D 점수 계산 후 **내림차순 정렬**.
2. 최대 점수 유형 → **Primary Type**
3. 2번째 유형 → **Secondary Type**
4. 동점 처리:

   * 동점이 있을 경우,

     * UI 상에서 “공동 1위 / 공동 2위”를 표기
     * 설명 텍스트는 2개 유형을 모두 보여줌.

```ts
type Result = {
  scores: { V: number; A: number; K: number; D: number };
  primary: 'V' | 'A' | 'K' | 'D'[];
  secondary: 'V' | 'A' | 'K' | 'D'[];
};
```

# 9. 결과 페이지(`/result`) 상세

## 9.1. 데이터 입력

* 프론트에서 `Result` 객체를 생성한 뒤,

  * URL state / router state를 통해 결과 페이지로 전달
  * or 전역 상태 관리 (Redux/Zustand 등) 사용
* 선택적으로 서버에 결과 POST (9.4. API 참고)

## 9.2. 구성 요소

1. **헤더**

   * "당신의 학습 전략 설계 결과"
   * 간단 서브타이틀:
     "점수는 성격을 '줄 세우기'가 아니라, **이런 방식으로 공부할 때 성과가 잘 나는 유형**을 보여줘요."

2. **V/A/K/D 점수 시각화**

   * Radar Chart or Bar Chart
   * 예: `chart.js` 또는 `recharts` 사용
   * 점수 labels, 툴팁(예: "시각형(V): 32점")

3. **주/보조 유형 요약**

   예시 텍스트 구조(카피는 이후 튜닝):

   * Primary: **시각형 (Visual)**

     * "이런 방식으로 공부할 때 성과가 잘 나는 유형입니다."
     * "정보를 눈으로 보고 이해하는 것을 가장 선호합니다."
   * Secondary: **논리형 (Digital/Logical)**

     * "자료를 분석하고 구조화하는 데 강점을 가진 유형입니다."

4. **유형별 상세 설명 블록**

   각 차원에 대해 공통 구조:

   * 제목: "V – 시각형(Visual)"
   * 3가지 섹션:

     1. **당신의 정보 처리 방식**
     2. **효과적인 학습법 추천**
     3. **추천 전공/진로 키워드 (가벼운 수준)**

   → v1에서는 간단 텍스트 하드코딩.
   (복수 유형일 경우 두 개 유형을 강조, 나머지는 "보조 경향"으로 간단 설명만)

5. **과목별 적용 가이드** (중요도 ★★★★★)

   일반 고등학생 결과지의 핵심 차별점.

   예시 구조:

   | 과목 | 이렇게 공부할 때 효과적 |
   | -- | ----------------- |
   | 국어 | (V) 지문 구조 시각화 / (A) 스스로 요약 설명 |
   | 수학 | (K) 손으로 풀이 반복 / (D) 풀이 절차 정리 |
   | 영어 | (A) 소리 내 읽기 / (V) 문장 패턴 표시 |
   | 탐구 | (D) 개념 간 연결도 / (V) 표·도식화 |

   → 유형별로 해당 과목에 맞는 전략을 구체적으로 제시.

6. **성적 정체 원인 분석**

   일반 고등학생은 **원인 분석**을 원합니다.

   예시:

   * Visual형인데
     → "문제 풀이를 **말 위주 설명**으로만 이해하려 했을 가능성"
   * Auditory형인데
     → "혼자 묵묵히 읽기만 했을 가능성"

   📌 포인트

   * 노력 부족 ❌
   * 지능 문제 ❌
   * **전략 불일치 ⭕**

7. **일반 고등학생 맞춤 솔루션 영역**

   각 학생에게 공통 + 유형별 TIP mix.

   예시 섹션 구조:

   * 🎓 "지금(내신/기본기) 이렇게 해보면 좋아요"
   * 📈 "앞으로(고3 대비/수능) 이 전략을 미리 준비하세요"
   * 🧠 "시험·과제 준비 공부법"
   * ⏰ "당신에게 맞는 하루 루틴 예시 (아침·오후·저녁)"
   * 🔧 "자기조절 학습 팁" (계획 세우기, 집중 회복, 복습 주기)

   → 텍스트는 유형별로 약간 달라짐 (조건부 렌더).
   → "지금 vs 앞으로" 전략을 명확히 구분.

6. **결과 공유/저장**

   * “결과 이미지로 저장하기” 버튼

     * 결과 카드 영역을 캡처(canvas or html-to-image 라이브러리 사용)
   * “링크 복사” 버튼 (단축 URL은 v2에서 고려)

7. **재검사 안내**

   * “나중에 다시 검사해보고 비교해보세요” 문구
   * [다시 검사하기] 버튼 → `/test` 리다이렉트

## 9.3. UI/UX 요구사항

* 모바일에서 스크롤이 너무 길지 않도록 **섹션 접기/펼치기** 고려
* 그래프는 터치에 최적화
* 눈에 잘 들어오는 색상(각 유형별 고유 컬러) 사용

백엔드/DB를 둔다면(예: Supabase, Firebase, PlanetScale 등):

```sql
Table: test_results
- id (uuid, pk)
- created_at (timestamp)
- scores_v (int)
- scores_a (int)
- scores_k (int)
- scores_d (int)
- primary_types (varchar)   -- 예: "V" or "V,A"
- secondary_types (varchar) -- 예: "D"
- client_type (varchar)     -- 'mobile' / 'desktop'
- user_agent (text, optional)
```

개인 식별 정보는 저장하지 않는다.

## 10.2. API (선택적 구현)

### POST `/api/results`

* 목적: 검사 완료 결과 저장
* Request Body (JSON):

```json
{
  "scores": { "V": 30, "A": 25, "K": 20, "D": 25 },
  "primaryTypes": ["V"],
  "secondaryTypes": ["A"],
  "clientType": "mobile"
}
```

* Response:

```json
{
  "id": "uuid",
  "createdAt": "2025-12-02T02:10:00.000Z"
}
```

### GET `/api/results/:id` (옵션)

* 특정 결과 다시 조회(링크 공유용)
* v1에서는 필수 아님. (기획자 판단)

# 15. 개발 시 주의할 점 (Cursor용 메모)

1. **문항·채점 로직은 위 테이블 기준으로 정확히 구현**

   * 각 문항에서 4/3/2/1 **중복 없이** 할당되어야 함.
2. **총점 검산**

   * `V + A + K + D === 100` 이 아닐 경우 버그로 판단하고 로그 남기기.
3. **타입 판정**

   * 동점 상황 고려(배열로 리턴).
4. **모바일 UX**

   * 4개의 진술문이 한 화면에 자연스럽게 보이도록 카드형 UI 구성.
5. **텍스트 상수 분리**

   * 문항/설명은 i18n 혹은 별도 JSON으로 분리하면 향후 수정이 용이.
6. **결과지 톤앤매너**

   * "너는 이런 애야" ❌
   * "이 상황에서는 이 전략이 낫다" ✅
   * 성취/효율/성과라는 단어는 허용하되, 서열·우열 표현은 금지.
7. **과목별 가이드 구현**

   * 국어, 수학, 영어, 탐구(사회/과학) 각 과목별로 유형에 맞는 구체적 전략 제시.
   * 표 형태로 명확하게 보여주는 것이 효과적.
